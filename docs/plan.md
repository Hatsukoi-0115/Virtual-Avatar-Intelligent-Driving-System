## Plan: Live2D多模态虚拟形象驱动系统

### 总目标
在 Windows 桌面端实现一套以 Live2D 为核心的多模态虚拟形象驱动系统，能够在直播场景下稳定接入摄像头、麦克风、实时语音识别、情绪理解和语义理解，并通过统一的 avatar controller 把这些信号融合后输出到 Live2D 角色表现。

### 核心判断
1. 这个项目可行，且最适合从 Windows-only MVP 开始。
2. 最大技术风险不在模型本身，而在 Live2D Python 封装、实时链路延迟和多源状态融合。
3. 系统必须采用“感知层 -> 决策层 -> 渲染层”的单向链路，避免多个模块直接修改头像状态。

### MVP 边界
1. 必做：桌面窗口、设置页、小窗预览、开始直播、结束直播、摄像头输入、麦克风输入、基础表情与动作驱动。
2. 必做：Live2D 小窗口能稳定打开、加载模型、更新参数。
3. 必做：MediaPipe 动作、FunASR 转写、情绪分类、LLM 语义理解都能接入统一控制层。
4. 暂不做：跨平台支持、复杂推流平台集成、编辑器级创作功能、平台化模型编排。

### 推荐实现顺序
1. 先验证 Live2D Python 封装是否可用，再做别的模块。
2. 再做桌面壳和应用状态机，保证产品操作闭环。
3. 然后接入视觉动作链路，优先完成最稳定、最高频的控制。
4. 再接入语音和情绪理解，补齐短时表达能力。
5. 最后接入 LLM 语义理解，并把它限制为低频修正。
6. 最终由 avatar controller 统一融合并输出到渲染层。

### 详细执行清单
#### 1. 技术预研与 POC
1.1 选择并验证 Live2D Python 封装是否能在 Windows 下完成窗口打开、模型加载、参数更新和基础表情切换。
1.2 验证摄像头采集链路能否稳定输出帧数据，确认分辨率、帧率和设备兼容性。
1.3 验证麦克风采集链路能否稳定输出音频流，确认采样率、缓冲长度和设备兼容性。
1.4 验证 MediaPipe 在目标机器上的实时性能，检查人脸关键点、头姿和嘴部特征是否稳定。
1.5 验证 FunASR 流式识别的部署方式，比较本地运行和本地优先加远端兜底两种路径。
1.6 验证 LLM 调用是否会显著拖慢主循环，明确其刷新频率和异步调用方式。
1.7 记录所有 POC 结果，形成是否继续使用当前 Live2D 封装的判断依据。

验收条件：能独立打开 Live2D 窗口并实时更新参数，同时确认视频、音频和模型调用不会明显卡死。

#### 2. 桌面应用骨架
2.1 搭建主窗口、设置页、小窗预览页和系统托盘。
2.2 建立应用启动流程、关闭流程和后台驻留流程。
2.3 设计直播状态机，至少包含未准备、准备中、运行中、停止中、错误恢复五个状态。
2.4 实现设备选择界面，包括摄像头、麦克风和渲染相关基础选项。
2.5 实现权限检查、设备不可用提示和恢复按钮。
2.6 实现配置持久化，保证重启后能恢复上次设备和基础偏好设置。
2.7 实现小窗置顶、显示隐藏和尺寸调整。

验收条件：用户可以完成启动、配置、开始、结束、退出的完整操作闭环，小窗可以独立稳定显示。

#### 3. 视觉动作链路
3.1 接入摄像头输入并建立稳定的视频采集线程或进程。
3.2 接入 MediaPipe，提取脸部关键点、头部姿态、眼睛开合和嘴部张合等特征。
3.3 定义动作特征结构，输出可直接用于头像驱动的连续值。
3.4 建立动作平滑、阈值过滤、异常值剔除和节流机制。
3.5 定义动作到 Live2D 参数的映射表，先覆盖头部、眼睛和嘴部等高频特征。
3.6 设计动作层刷新频率，避免阻塞桌面主线程。
3.7 编写动作驱动回放测试，验证长时间运行不抖动、不漂移。

验收条件：视觉动作能稳定驱动角色表现，且不会明显卡顿、跳变或阻塞主界面。

#### 4. 语音识别与情绪理解
4.1 接入麦克风输入并建立音频缓存、分段和回调机制。
4.2 接入 FunASR 流式识别，输出增量文本和稳定文本。
4.3 建立断句、去噪和文本稳定化机制，避免把碎片化结果直接交给上层。
4.4 设计情绪分类策略，可先使用轻量模型或规则辅助方案。
4.5 输出情绪标签和置信度，至少覆盖开心、难过、愤怒、平静等基础类型。
4.6 将情绪结果送入 avatar controller，但保持低于动作层的优先级。
4.7 编写语音到情绪的测试样本，验证结果变化是否合理且不过度抖动。

验收条件：语音输入能稳定转写，情绪变化能驱动表情变化，但不会频繁来回切换。

#### 5. LLM 语义理解
5.1 定义 LLM 的输入格式，只接收较长文本片段、当前上下文和必要状态。
5.2 定义 LLM 的输出格式，只返回可解释的语义状态或慢变化建议。
5.3 设置低频刷新策略，避免 LLM 进入高频主循环。
5.4 设计异步调用和超时回退，避免阻塞渲染和识别链路。
5.5 将语义输出转成慢变化的表情、姿态或待机行为。
5.6 设计与短时情绪的冲突消解策略，保证语义只做修正不做抢占。
5.7 测试长句、停顿和上下文切换场景，检查语义变化是否合理。

验收条件：LLM 结果不会影响实时主循环，只在较长时间尺度上改变角色状态。

#### 6. avatar controller 融合层
6.1 设计统一状态对象，区分动作、表情、语义、设备状态和错误状态。
6.2 定义优先级规则，明确动作、情绪和语义在冲突时的覆盖顺序。
6.3 定义持续时间规则、冷却时间和状态过期机制。
6.4 定义状态平滑策略，防止短时间内频繁切换。
6.5 定义异常回退策略，在某一输入源失效时保持角色可用。
6.6 定义输出到 Live2D 渲染层的统一指令格式。
6.7 编写融合策略测试，覆盖同时存在动作、情绪和语义的冲突场景。

验收条件：三类输入同时存在时，系统输出仍可解释、可预测，并且不会出现表情抖动。

#### 7. Live2D 渲染集成
7.1 选择最终采用的 Live2D Python 封装方案，并锁定接口边界。
7.2 定义渲染层最小职责，只负责显示、参数更新和基础状态展示。
7.3 建立头像模型加载、重载和切换流程。
7.4 建立参数更新接口，把头像状态映射为渲染层调用。
7.5 建立渲染错误处理和恢复机制。
7.6 预留替代渲染实现接口，以便后续替换封装方案。

验收条件：Live2D 渲染层可以独立工作，也可以接入融合层稳定更新参数。

#### 8. 稳定性与发布准备
8.1 做长时间运行测试，检查是否有内存泄漏、句柄泄漏或线程失控。
8.2 做性能测试，记录 CPU、内存、GPU 和帧率占用。
8.3 做设备断连测试，确认摄像头或麦克风异常时能否恢复。
8.4 做权限失败测试，确认用户能看懂错误提示并重试。
8.5 做模型加载失败测试，确认应用能降级而不是直接崩溃。
8.6 优化启动时间和切换流程，减少直播准备时间。
8.7 做打包和发布脚本，形成可复现的部署流程。

验收条件：系统能够稳定连续运行，发生异常时可以恢复，且用户启动和切换体验不明显卡顿。

### 依赖关系
1. POC 先于所有正式开发，因为它决定 Live2D 封装和部署路线。
2. 桌面骨架先于各感知模块，因为它提供应用状态和 UI 宿主。
3. 视觉动作先于语音和语义，因为它是最实时、最稳定的第一条驱动链路。
4. 语音情绪先于 LLM 语义，因为短时情绪更适合作为中间层。
5. avatar controller 必须在多模态输入全部接入前完成基本结构设计。
6. 稳定性测试必须贯穿后半程，而不是最后才补。

### 风险与应对
1. Live2D Python 封装不稳定：先 POC，必要时保留替代渲染实现。
2. 实时延迟过高：采集、推理、渲染分线程或分进程隔离。
3. 状态抖动：统一做节流、平滑和冲突消解。
4. 模型部署复杂：首版优先本地可跑，云端只做兜底。
5. 复杂度膨胀：严格控制 MVP，优先完成完整链路而不是堆功能。

### 交付物
1. 可运行的 Windows 桌面应用骨架。
2. 可独立验证的 Live2D 驱动 POC。
3. 从摄像头、麦克风到 avatar controller 的统一状态链路。
4. 明确的模块接口、状态定义和优先级规则。
5. 一套可执行的测试与验收清单。
